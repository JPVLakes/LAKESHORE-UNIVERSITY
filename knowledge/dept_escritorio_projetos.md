## Processos

### Processo 1: Desenvolvimento de Projeto Pre-Contrato (Custom Homes)

- **Objetivo:** Desenvolver planta arquitetonica personalizada para cliente custom, desde o primeiro contato ate a assinatura do contrato de construcao
- **Responsavel principal:** Camila Martins
- **Frequencia:** A cada novo cliente custom
- **Ferramentas:** AutoCAD, SketchUp, Google Drive, ClickUp (tracking interno)

**Passo a passo:**
1. Cliente entra em contato (diretamente com Camila, Bernardo ou Juliana) -- Responsavel: Camila -- Ferramenta: E-mail/WhatsApp
2. Meeting com cliente para solicitar projeto customizado -- Responsavel: Camila -- Ferramenta: [presencial/video]
3. **Tracking inicia aqui** - Dia 1 do projeto -- Responsavel: Camila -- Ferramenta: ClickUp
4. Camila desenvolve o projeto (planta arquitetonica) -- Responsavel: Camila -- Ferramenta: AutoCAD
5. Apresentacao da proposta ao cliente (~1 semana) -- Responsavel: Camila -- Ferramenta: [presencial/video]
6. Se cliente aprova: apresentar proposta de **pre-agreement** (valor cobrado para desenvolver a planta completa) -- Responsavel: Camila -- Ferramenta: E-mail
7. Trabalho na planta: parte estrutural, elevacoes, 3Ds -- Responsavel: Equipe de arquitetura -- Ferramenta: AutoCAD, SketchUp, Enscape
8. Enviar projeto para engenheiro -> retorna projeto executivo (PDF) -- Responsavel: Luana -- Ferramenta: E-mail
9. Projeto vai para setor de **Budget** (Samuel e equipe) -- Responsavel: Camila -- Ferramenta: ClickUp/E-mail
10. Budget: takeoffs para criar orcamento -- Responsavel: Samuel/equipe Budget -- Ferramenta: Planilhas
11. Budget volta para Camila -> reuniao com cliente -- Responsavel: Camila -- Ferramenta: [presencial/video]
12. Cliente aprova budget -> solicitar **contrato de construcao** -- Responsavel: Camila -- Ferramenta: E-mail

**Inputs:** Solicitacao do cliente, dados do terreno, preferencias do cliente
**Outputs:** Planta aprovada, budget aprovado, contrato de construcao solicitado
**Observacoes:** Nao existe tracking formal para pre-contrato atualmente. Clientes customizados mudam muito (acabamentos, pisos, cabinets) -- budget nao e definitivo. Budget customizado apresentado condensado ao cliente (flooring como categoria, nao item por item).

### Processo 3: Take-Off de Materiais

- **Objetivo:** Quantificar materiais necessarios para construcao a partir dos projetos executivos
- **Responsavel principal:** Gabriela Medeiros / Candida Tenorio
- **Frequencia:** A cada novo modelo (spec) ou projeto (custom)
- **Ferramentas:** BlueBeam, Word, Google Drive

**Passo a passo:**
1. Receber projeto executivo (PDF) do engenheiro -- Responsavel: Luana -- Ferramenta: E-mail
2. Abrir PDF no BlueBeam -- Responsavel: Gabriela/Candida -- Ferramenta: BlueBeam
3. **Calibracao obrigatoria**: colocar escala correta usando cotas de referencia -- Responsavel: Gabriela/Candida -- Ferramenta: BlueBeam
4. Tracar areas, linhas e pontos para quantificar materiais:
   - Volume de concreto de lajes e furers
   - Quantitativo de blocos (CMU/lintels)
   - Paredes (drywall)
   - Pisos e revestimentos
   - Bancadas (countertops)
   - Iluminacao pontual
   - Plumbing fixtures
   - Portas e janelas (com direcao de abertura)
   - Acabamentos externos (stucco, painting)
   - Telhado (com angulacao para calcular area de shingles)
   - Trims, mao francesa (brackets)
   -- Responsavel: Gabriela/Candida -- Ferramenta: BlueBeam
5. Compilar resultados **manualmente** em lista no Word -- Responsavel: Gabriela/Candida -- Ferramenta: Word
6. Adicionar 10% de desperdicio -- Responsavel: Gabriela/Candida -- Ferramenta: Word
7. Salvar PDF do takeoff no Drive em pasta acessivel -- Responsavel: Gabriela/Candida -- Ferramenta: Google Drive
8. Para revisoes: arquivo antigo vai para pasta "revision", novo substitui -- Responsavel: Gabriela/Candida -- Ferramenta: Google Drive

**Inputs:** Projeto executivo em PDF (do engenheiro)
**Outputs:** Lista de quantidades de materiais por item
**Observacoes:** Compilacao e totalmente manual do BlueBeam para Word. Funcao paga do BlueBeam para linkar com planilhas nunca foi utilizada. Spec homes: takeoff feito uma vez por modelo. Custom homes: cada projeto requer takeoff unico. Possibilidade futura de APIs do BlueBeam para puxar dados automaticamente.

- Budget customizado feito "no escuro" com estimativas de area e nivel no estagio inicial
- Mudancas tardias exigem refazer budgets (ex: Bernardo pediu incluir fee em 18 budgets)
- **CPUs da empresa nao rodam programas de arquitetura** (AutoCAD, SketchUp, Enscape) - insuficientes
- Problemas de instalacao/licenca do AutoCAD para o setor
- Compilacao manual do takeoff do BlueBeam para lista no Word (processo lento e propenso a erros)
- **3.500 itens** para atualizar manualmente entre modelos similares (falta template/duplicacao)
- Parte de acabamentos usa allowances em vez de valores fixos
- Equipe trabalha muito isolada internamente, pouca interacao com outros setores

## Takeoff Integration

### How Project Office Sends Design Docs to Takeoff
```
Design document ready for takeoff
  |
  +-> Luana receives project executivo (PDF) from engineer
  |
  +-> Save to Google Drive in project-specific folder
  |     - Folder structure: /administration/{project_name}/
  |     - PDF format with correct scales for BlueBeam calibration
  |
  +-> Notify Takeoff team (Gabriela Medeiros / Candida Tenorio):
  |     - Project name and model type (spec or custom)
  |     - Location of PDF in Drive
  |     - Spec sheet references (Level 1, 2, 3, or 4 for spec homes)
  |     - Design book (for custom homes)
  |     - Any special notes (non-standard materials, custom finishes)
  |
  +-> Takeoff team performs material quantification in BlueBeam:
        - Calibrate scale using reference dimensions
        - Trace areas, lines, and points for all material categories
        - Compile quantities manually into Word document
        - Add 10% waste factor
        - Save takeoff PDF to Drive
```

### Revision Workflow for Plan Changes
```
Plan change identified (by client, engineer, or field)
  |
  +-> Notify Takeoff team of affected items:
  |     - List specific changes (dimensions, materials, layout)
  |     - Identify which takeoff categories are impacted
  |
  +-> Takeoff team re-quantifies affected items ONLY:
  |     - Old takeoff moved to "revision" subfolder
  |     - New takeoff replaces old in active folder
  |
  +-> Quantities delta calculated:
        - More material needed -> notify Compras for additional PO
        - Less material needed -> notify Compras to adjust/cancel PO
        - Same material, different spec -> notify Compras of substitution
```

### Key Takeoff Categories
Concrete (slabs, footers), CMU/lintels, drywall, flooring, countertops, lighting, plumbing fixtures, doors/windows (with opening direction), exterior finish (stucco, paint), roofing (with angle calculation for shingle area), trims, brackets

## Revision Workflow

### Client-Requested Changes
```
Client requests design change
  |
  +-> Camila assesses feasibility:
  |     - Can it be accommodated at current construction stage?
  |     - Does it require engineering recalculation?
  |     - Does it affect permits (structural changes)?
  |
  +-> IF feasible:
  |     +-> Update plans in AutoCAD / SketchUp
  |     +-> Version old plans as "superseded" in Google Drive
  |     +-> Notify ALL affected departments:
  |           - Takeoff: re-quantify affected items
  |           - Orcamento: revise budget with new quantities
  |           - Scheduling: assess timeline impact
  |           - Permits: IF structural change -> may need permit revision
  |           - Compras: IF materials change -> adjust POs
  |           - Field/Supervisores: distribute updated plans for construction
  |
  +-> IF not feasible:
        - Document reason (structural limitation, code violation, cost prohibitive)
        - Present alternatives to client via Client Relations
        - Client decides: accept alternative, withdraw request, or escalate to Camila/Bernardo
```

## Project Development Detail (Batch 3)

### Project Lifecycle Stages (from Camila Martins interview, 21/05/2025)

The full project lifecycle from initial client contact to construction handoff follows these stages:

1. **Client Contact** -- Client reaches Camila, Bernardo, or Juliana directly
2. **Concept Meeting** -- Meeting to define custom project requirements; tracking starts (Day 1)
3. **Design Development** -- Camila develops architectural plan (~1 week to first proposal)
4. **Pre-Agreement** -- Value charged to develop the full plan; client commits financially before full design
5. **Full Design** -- Structural plans, elevations, 3D renderings produced by architecture team
6. **Engineering** -- Project sent to external engineer; returns as executive project (PDF)
7. **Budget Creation** -- Samuel/Budget team receives plans for takeoff and costing
8. **Client Approval** -- Camila presents budget to client in condensed format (categories, not line items)
9. **Contract Signing** -- Client approves budget; construction contract requested
10. **Execution Handoff** -- Cris receives signed contract + all documents for construction start

**Key distinction:** Spec homes follow a simplified flow (strategic meeting -> design -> standard budget by county -> released for sale). Custom homes require individual design, pre-agreement, and unique budget.

### Design Book Assembly Process

The **design book** is a critical output document assembled by the architecture team:
- **Contents:** Floor plans, elevations, specification sheets, 3D renderings, material selections
- **Assembled by:** Camila's team (Luana coordinates, Gabriela/Candida/Henrique contribute)
- **Used by:** Budget (for costing), Permits (for submission), Field (for construction), Client Relations (for client portal)
- **Spec homes:** Standardized design books per model; variations only for elevations (e.g., farm vs contemporary facade)
- **Custom homes:** Unique design book per project; evolves as client changes selections
- **Storage:** Google Drive in "administration" folder; PDFs attached to contract package

### Client Revision Workflow and Downstream Impact

Client revisions during active construction are a major pain point with cascading effects:

1. **Revision Request** -- Client contacts Camila or Client Relations with change (acabamentos, pisos, cabinets, stone veneer, pavers)
2. **Feasibility Check** -- Camila assesses: is it structurally possible? Does it need engineering? Does it affect permits?
3. **Impact Assessment** -- If approved:
   - **Takeoff:** Must re-quantify affected material categories
   - **Budget/Orcamento:** Must update cost estimates (custom budgets are a moving target)
   - **Scheduling:** May require rescheduling trades (e.g., change from vinyl to tile delays flooring phase)
   - **Permits:** Structural changes may require permit modification (hold risk)
   - **Compras/Purchasing:** POs may need adjustment or cancellation
   - **Field:** Supervisors need updated plans immediately
4. **Communication Gap** -- Currently, revision notifications are manual and often delayed. Field may build with outdated plans.

**Batch 3 insight:** Camila noted that allowances are used instead of fixed values for finishing items in custom budgets. The budget presented to clients is condensed (e.g., "flooring" as a category, not individual items). This creates a gap between what the client sees and what the budget team tracks internally.

### Cases Requiring Special Handling
- **HOA (Home Owners Association):** Some regions require HOA approval before permit application
- **Demolitions:** Require specific demolition permit BEFORE construction permit
- **Renovations:** Also require permits; different process from new construction
- **Commercial projects:** More complex than residential; not standard flow
- **Land Development:** Separate lifecycle (project, permit, execution, closing) managed by Vinicius
